# Migration safety

How Switchboard schema changes ship without downtime or data loss. Read this
before authoring any migration.

## Where migrations live and how they run

- **Source of truth:** `apps/api/src/db/migrations/*.sql` + the Drizzle journal
  `apps/api/src/db/migrations/meta/_journal.json`. Generated by `drizzle-kit
generate` from `apps/api/src/db/schema.ts`.
- **Applied in production** by `deploy/scripts/migrate.mjs` on api boot (the
  `server` role, when `MIGRATE_ON_BOOT=1`), using drizzle-orm's node-postgres
  migrator. No `drizzle-kit` is needed in the runtime image — `drizzle-orm` + `pg`
  are already production deps.
- **Applied in tests** by `apps/api/src/db/test-helpers.ts` on PGlite. It is the
  **same journal**, so every migration is exercised on real Postgres semantics in
  CI before it ever reaches a server. One migration path, two callers.
- **Ordering:** drizzle applies migrations in **journal `idx` order** (0,1,2,…),
  keyed by tag + hash in the `__drizzle_migrations` table. Re-running is a no-op —
  applied tags are skipped.

## The 0010 / 0011 journal story (why the numbers jump)

The migration tags are, in journal order:

```
idx 0  0000_bootstrap              (citext extension)
idx 1  0001_c1_schema              (the C1 domain schema)
idx 2  0002_c1c_indexes            (§9 hot-column + activity indexes)
idx 3  0003_c1e_search             (pg_trgm + FTS/trigram indexes)
idx 4  0010_imports               (CSV import job state — task 4f, D-018)
idx 5  0011_audit_log_append_only (audit_log UPDATE/DELETE/TRUNCATE triggers — 5b)
```

The jump from `0003` to `0010` is **intentional and correct**, not a gap of
missing migrations. Under the max-parallel build policy (DECISIONS **D-017**),
migration numbers were **reserved per stream** so concurrent agents in separate
worktrees could add migrations without colliding: the CSV-import stream reserved
`0010`, the admin-ops/audit stream reserved `0011`. Each landed only its reserved
number.

Consequences that matter:

- **`idx` is contiguous (0–5); the numeric tags are not.** drizzle cares only
  about `idx` order + tag/hash, so the numeric gap is pure bookkeeping.
- **Never renumber to "close" the gap.** Renaming a tag changes its hash;
  `__drizzle_migrations` would then re-run an already-applied migration and fail.
- **The next migration is `0012_…`** (continue the max reserved number forward),
  regardless of the `0004`–`0009` gap.

## Additive-first rules

Default to changes that are safe against the _currently running_ app (the old
image is still serving during a rolling deploy, and is the rollback target):

**Safe (ship in one migration):**

- `CREATE TABLE`, `CREATE INDEX`, `CREATE EXTENSION IF NOT EXISTS`.
- `ADD COLUMN` that is nullable or has a constant `DEFAULT`.
- `ADD CONSTRAINT … NOT VALID`, then `VALIDATE CONSTRAINT` in a later migration.
- New enum **values** (append only).

**Unsafe in a single step — use expand→migrate→contract below:**

- `DROP COLUMN` / `DROP TABLE` / `RENAME` (old image references the old name).
- `ALTER COLUMN … TYPE` that narrows or rewrites.
- `SET NOT NULL` on a populated column without a prior backfill + default.
- Removing or renaming enum values.

**Index note:** the existing index migrations (`0002`, `0003`) use plain
`CREATE INDEX`, which takes a brief write lock — fine at v1 scale. `CREATE INDEX
CONCURRENTLY` **cannot run inside a transaction**, and the migrator wraps each
migration in one, so a concurrent build must be applied out-of-band (documented,
manual) rather than as a normal journaled migration. Measure before reaching for
it.

## Shipping a destructive change safely — expand · migrate · contract

Three independently-deployable steps, each rollback-safe:

1. **Expand.** Add the new shape additively (new column/table). Backfill existing
   rows. Teach the app to write BOTH old and new. Deploy. Old and new coexist; the
   previous image still works.
2. **Migrate.** Switch the app's reads to the new shape; backfill any rows written
   between steps. Deploy. The old shape is now unused but still present.
3. **Contract.** Take a backup (`deploy/scripts/backup.sh`), confirm no code
   references the old shape, then drop it. Deploy.

Because every step is additive until the final drop, a rollback at any point is
just redeploying the previous image against a forward-compatible database.

## Multi-writer / multi-replica honesty (v1 = single writer)

`migrate.mjs` takes a **session-level Postgres advisory lock** on its one
connection, so two booting containers serialise (the second blocks, then finds
nothing to apply). On top of that, only the **`server`** role migrates
(`MIGRATE_ON_BOOT=1`); the `worker` role and any extra api replicas set
`MIGRATE_ON_BOOT=0`.

For v1 the assumption is **one migrating writer**. If you scale the api to
multiple replicas, do NOT let them all migrate on boot — run migrations as a
**dedicated one-shot** before rolling the fleet:

```bash
docker compose run --rm -e APP_ROLE=server -e MIGRATE_ON_BOOT=1 api node /app/apps/api/migrate.mjs
# then roll the api replicas with MIGRATE_ON_BOOT=0
```

The advisory lock makes even a botched concurrent boot safe, but the one-shot is
the intended path.

## Rollback = restore (there are no down-migrations)

This project ships **no down migrations** by design. To undo a bad schema change:

1. Redeploy the **previous image tag** (the app half of the rollback).
2. If the schema change was destructive and the new image already dropped/rewrote
   data, **restore from the pre-upgrade backup** (`deploy/scripts/restore.sh` is
   the _drill_; the real overwrite-prod restore is the runbook in
   `deploy/README.md`).

This is why the upgrade runbook takes a backup **before** every deploy: the backup
IS the rollback for anything the expand→migrate→contract discipline can't make
reversible.
