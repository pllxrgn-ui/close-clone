# Switchboard `apps/web` — architecture audit (2026-07-17)

Senior-engineer-style reverse-engineering pass over the web app: full data-flow map,
ranked problem areas, functionality-preserving refactoring strategies, and code
sketches. Produced by a read-only review agent; verified findings are marked
CONFIRMED (code path traced). Nothing in this report was applied without a
separate reviewed change.

## A. The system as it IS

**Entry & boot.** `src/main.tsx` imports global CSS (`tokens.css` → `base.css` →
`primitives.css` → `shell.css` → `overlays.css`), then `enableMocking()` gates the
mock layer purely on `import.meta.env.VITE_API_MODE`: any value other than
`"real"` dynamically imports `src/mocks/browser.ts` and starts the MSW service
worker with `onUnhandledRequest:'bypass'`. Only after that promise resolves does
it render. `"real"` mode never loads the MSW chunk and talks to the API through
the Vite `/api` proxy. This is the single dual-mode seam and it is clean — no
product code below `main.tsx` branches on the mode.

**Provider stack.** `App.tsx` → `AppProviders` nests `ThemeProvider` →
`QueryClientProvider` → `AuthProvider`, router-agnostic so tests mount it around
a `MemoryRouter`. `createQueryClient` (`src/app/queryClient.ts`): `staleTime 30s`,
`gcTime 5m`, `refetchOnWindowFocus:false`, retries 5xx twice, never retries 4xx.
Feedback (`ToastProvider`), Keyboard, and Command (palette) providers wrap the
shell.

**Router & auth gate.** `src/app/AppRoutes.tsx` is fully code-split — every page
is `lazy()`. `/login` and `/welcome` are public; everything else nests under
`RequireAuth` → `AppShell`. Auth is a mock stub hydrating a fixture `User`
synchronously from `localStorage` (no boot flash); real OIDC replaces the innards
without touching the context shape.

**Feature pages → api clients → MSW.** Each feature owns `api/` typed wrappers
over `apiRequest` (`src/api/client.ts` — base `/api/v1`, typed `ApiError` from the
C8 envelope, keyset `{items,nextCursor}`). Reads are `useQuery`/`useInfiniteQuery`;
writes are `useMutation`; the pipeline board does optimistic cache updates. No
WebSocket client exists in the web app — REST + query invalidation only.

**MSW composition (the load-bearing seam).** `src/mocks/browser.ts` and
`server.ts` compose in the same order: core `handlers` → `inboxHandlers` →
`pipelineHandlers` → `adminHandlers` → `reportsHandlers` → `commsHandlers` →
`viewBuilderHandlers` → `leadDetailHandlers`. MSW is first-match-wins; several
routes are multiply-owned and resolved cooperatively (see problem 5).

**Dependency layering.** `ui/` depends only on `lib/` + tokens (verified: no
`ui/**` file imports a feature). `api/` depends on `@switchboard/shared` only.
`keyboard/`, `command/`, `feedback/`, `theme/`, `auth/` are cross-cutting leaves.
`app/` depends downward on everything. `features/*` depend on the leaves — the
"features never import features" rule is violated laterally (problem 6).

**State inventory.** Query keys: pipeline `['pipeline',*]`; leads `['leads','all']`,
`['ref',*]`, `['smart-views']`, `['smart-view',id]`, `['smart-view-preview',…]`;
reports `['reference','users']`; view-builder `['users']` — note four different
keys for the same `/users` (problem 4). Module singletons:
`pipeline/state/boardInteraction.ts` (palette bridge observable) + the per-feature
MSW demo stores. Contexts: Theme, Auth, Keyboard, Toast, Comms. Web storage:
auth + theme in `localStorage`, welcome ignition flag in `sessionStorage` — all
keyed and bounded.

## B. Critical problem areas (ranked by impact × likelihood)

### 1. Pipeline drains ALL leads to build a name map — `features/pipeline/api/opportunities.ts:40-51` · CONFIRMED

`fetchLeadNames` walks `GET /leads` to cursor exhaustion (page 200) to build a
`leadId→name` map, because opportunities carry no lead label. At 100k leads that
is ~500 sequential round-trips per board mount — fetching every lead in the org
to label the cards on screen. The sibling `fetchAllOpportunities` drain IS
justified (exact money totals need every deal). This is the single worst
scalability bug in the app. _(Fix in flight at time of writing.)_

### 2. Every board move re-drains the entire pipeline — `features/pipeline/components/PipelineBoard.tsx:174-177` · CONFIRMED

The move mutation is optimistic (correct), but `onSettled` invalidates
`OPPS_KEY`, whose queryFn is the full-cursor drain. A rep flicking deals with the
keyboard — the headline interaction — triggers a full-board refetch per
keystroke. The invalidation buys nothing: the server confirms the single row the
optimistic update already wrote.

### 3. Client-side sort/filter over keyset-paginated leads — `features/leads/components/LeadsSurface.tsx:113-130` · CONFIRMED

"All leads" loads pages of 100, then sorts and text-filters only the loaded
window while the header advertises `aria-sort` as if global. "Sort by value
desc" shows the top of the loaded window, not the true top; the filter silently
misses unfetched matches. Quietly wrong past a few hundred leads.

### 4. Four cache keys for one `/users` resource · CONFIRMED

`['ref','users']` (leads, comms), `['pipeline','users']`, `['reference','users']`
(reports), `['users']` (view-builder). Up to four identical fetches + four cache
entries with unshared invalidation. Same minor drift for lead-statuses/stages.

### 5. `GET /sequences` is a dead shadow held together by an id-alignment comment — `comms/mocks/commsHandlers.ts:153` vs `admin/mocks/adminHandlers.ts:48` · CONFIRMED

Both register `GET /sequences`; admin composes first and always wins — the comms
handler is unreachable. Correctness survives only because `commsStore` seeds
sequence ids to mirror `adminStore` (enforced by a comment, not a test). History
already records this contract breaking once ("every sequence showed 0 steps").
The other multiply-owned routes are genuinely collision-safe (cooperative
fall-through with cloned-body probing + production-order tests).

### 6. Lateral feature→feature coupling — `leads/components/LeadHeader.tsx:5`, `leads/components/BulkBar.tsx:4`, `comms/data/store.ts:34` · CONFIRMED

`leads` imports product components from `comms` and `admin`; `comms`'s mock store
imports `admin`'s. Real composition needs, wrong boundary — they defeat part of
the route-level code splitting and create a cycle risk. Fix is a shared boundary
(host-level composition), not deletion.

### 7. Duplicated helpers and parallel state registries · CONFIRMED

`initials()` vs `monogram()`; three money formatters (leads, reports, pipeline);
two relative-time helpers with different phrasing; the six-state vocabulary
modeled three times with diverging keys (`reply` vs `newReply`, `seq` vs
`inSequence`) across `ui/Lamp.tsx`, `leads/lib/leadState.ts`,
`welcome/fixtures.ts`; eight hand-rolled per-feature icon sets. Broad but
low-severity: every rename/precision change is a multi-file edit and the
state-key divergence is an active drift hazard.

### 8. `ownerNameOf` does `users.find` per card — `PipelineBoard.tsx:95-101` · PLAUSIBLE (low)

Team-sized users list × 30-card render cap = trivial today; flagged for
inconsistency with `leadNameOf`'s Map.

## C. Refactoring strategies (top 5 — all functionality-preserving)

1. **Bounded lead-name resolution.** Keep `fetchLeadNames`'s Map contract; source
   names only for leadIds present on the board via a batch `GET /leads?ids=…`
   (additive to C7, mirrored in the mock). Safety net first: a test with a
   fixture where most leads have no opp, asserting every card labels correctly
   with no over-fetch.
2. **Stop re-draining on move.** Delete the `onSettled` invalidation; reconcile
   the confirmed row in `onSuccess` (rollback already exists in `onError`).
   Safety net: N moves ⇒ exactly N PATCHes and zero `GET /opportunities`
   refetches.
3. **Honest lead sort/filter.** Push `sort`/`q` into the query key + params so
   the server orders/filters (mock taught to match); or, if out of scope, scope
   the UI honestly (disable header sort past page one). Safety net: a >1-page
   fixture asserting "value desc" surfaces the global max — this test fails
   today and encodes the bug.
4. **One reference-query module.** `src/api/refQueries.ts` exporting
   `queryOptions` for users/lead-statuses/opportunity-stages under `['ref',*]`;
   replace the four call sites mechanically.
5. **Collapse duplicated helpers + unify state keys.** Single home in
   `src/lib/format.ts` with re-export shims first (zero call-site churn), then
   migrate and delete shims. Derive `leadState` keys from `ui/Lamp.tsx`'s
   `LampState` with a mapping for the legacy names. Safety net: characterization
   tests pinning current outputs before merging implementations.

## D. Improved-code sketches

**#1 — resolve only what's on the board:**

```ts
export async function fetchLeadNames(
  leadIds: Iterable<string>,
  signal?: AbortSignal,
): Promise<Map<string, string>> {
  const ids = [...new Set(leadIds)];
  const names = new Map<string, string>();
  for (let i = 0; i < ids.length; i += 200) {
    const batch = ids.slice(i, i + 200);
    const page = await apiRequest<Page<Lead>>('/leads', {
      query: { ids: batch.join(',') },
      ...(signal ? { signal } : {}),
    });
    for (const l of page.items) names.set(l.id, l.name);
  }
  return names;
}
```

**#2 — reconcile the one row, never re-drain:**

```ts
const moveMutation = useMutation({
  mutationFn: (v: MoveVars) => moveOpportunity(v.id, { stageId: v.stageId, status: v.status }),
  onMutate: async (v) => {
    /* optimistic write + snapshot (unchanged) */
  },
  onError: (_e, _v, ctx) => {
    if (ctx?.prev) qc.setQueryData(OPPS_KEY, ctx.prev);
  },
  onSuccess: (server) => {
    qc.setQueryData<Opportunity[]>(OPPS_KEY, (old) =>
      old?.map((o) => (o.id === server.id ? server : o)),

  },
  // no onSettled invalidation — optimistic write + confirmed row are the truth
});
```

```ts
// src/api/refQueries.ts
import { queryOptions } from '@tanstack/react-query';
import { listUsers, listLeadStatuses, listOpportunityStages } from './reference.ts';

export const usersQuery = () =>
  queryOptions({ queryKey: ['ref', 'users'] as const, queryFn: () => listUsers() });

  queryOptions({ queryKey: ['ref', 'lead-statuses'] as const, queryFn: () => listLeadStatuses() });

  queryOptions({

    queryFn: () => listOpportunityStages(),
  });
```

## Checked and found HEALTHY

- The dual mock/real seam: `main.tsx` is the only mode branch; every client is
  mode-agnostic; MSW mirrors the C7/C8 shapes exactly.
- Cooperative MSW collisions (`enroll`, `emails/send`, `sms/send`,
  `tasks/:id`, `/opportunities`, `/opportunity-stages`): cloned-body probing +
  fall-through, guarded by production-order tests (`routeCollisions.*`) that
  refuse `server.use` — the D-029 lesson codified. Only `GET /sequences` is a
  dead shadow (problem 5).
- Leads table is virtualized (`@tanstack/react-virtual`, overscan 12, keyset
  prefetch near the tail).
- Route-level code splitting is real (every page a distinct `lazy()`), modulo
  problem 6.
- Query defaults sensible; mutations don't retry; 4xx never retried.
- Pipeline bounded DOM rendering (cap 30 + pinned active card) keeps money math
  full-data while capping DOM.
- Optimistic bulk-lead patching (`admin/bulk/leadCache.ts`) is
  reference-preserving with rollback; O(cache) walk is a defensible trade.
- Reports are server-aggregated with keyset pagination — no client joins.
- Web storage bounded and keyed; no growth vectors.
- Compliance rails enforced in the mock handlers server-side (422 SUPPRESSED on
  DNC; recording refusals; audited reasons) — I-RAIL-API holds in the demo too.
